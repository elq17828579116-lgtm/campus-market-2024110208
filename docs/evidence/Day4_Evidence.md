## 一、发布表单设计

### 1. 发布类型设计

本日支持四类业务发布，用户通过顶部标签切换：

| 发布类型 | 对应数据集合 | 关键字段 |
|---------|-------------|---------|
| 二手交易 | trades | 标题、分类、价格、成色、地点、联系方式、描述 |
| 失物招领 | lostFounds | 标题、类型（丢失/拾到）、物品名称、地点、发生时间、联系方式、描述 |
| 拼单搭子 | groupBuys | 标题、目标人数、截止时间、地点、联系方式、描述 |
| 跑腿委托 | errands | 标题、任务类型、酬劳、取件地点、送达地点、截止时间、加急、联系方式、描述 |

### 2. 字段设计说明

- **标题**：所有类型共用，必填，50 字以内
- **分类/类型**：各业务特有字段，使用 `<select>` 下拉或 `<radio>` 单选
  - 二手交易：需要分类（电子产品/教材书籍等）和成色（全新/几乎全新等）
  - 失物招领：需要区分"丢失"/"拾到"，以及具体的物品名称
  - 拼单搭子：需要目标人数（至少 2 人）、截止时间
  - 跑腿委托：需要任务类型、酬劳、取件/送达地点、加急选项
- **地点**：二手交易、失物招领、拼单搭子共用；跑腿委托拆分为取件/送达两个字段
- **联系方式**：所有类型共用，必填
- **描述**：所有类型共用，必填，500 字以内

### 3. 组件化设计

使用 `FormField.vue` 组件统一管理表单项的标签、必填标识和错误提示，避免重复的模板代码。

```vue
<FormField label="标题" required :error="errors.title">
  <input v-model="form.title" class="form-input" />
</FormField>
```

## 二、表单校验设计

### 校验规则

| 字段 | 规则 | 说明 |
|------|------|------|
| title | 非空 | 所有类型必填 |
| category | 非空 | 仅二手交易 |
| price | > 0 | 仅二手交易 |
| condition | 非空 | 仅二手交易 |
| itemName | 非空 | 仅失物招领 |
| targetCount | >= 2 | 仅拼单搭子 |
| deadline | 非空 | 拼单搭子、跑腿委托 |
| taskType | 非空 | 仅跑腿委托 |
| reward | > 0 | 仅跑腿委托 |
| from | 非空 | 仅跑腿委托 |
| to | 非空 | 仅跑腿委托 |
| contact | 非空 | 所有类型必填 |
| description | 非空 | 所有类型必填 |

### 校验逻辑

使用 `validate()` 函数统一校验，返回布尔值，同时将错误信息写入 `errors` 响应式对象。表单字段通过 `FormField` 组件的 `:error` prop 实时显示错误信息。

## 三、数据提交与页面联动

### 提交流程

1. 用户填写表单 → 点击"立即发布"
2. `handleSubmit()` 触发 `validate()` 校验
3. 校验通过 → 根据 `type` 调用对应的 `addXxx()` API 方法
4. 使用 Axios 向 JSON Server 发送 POST 请求
5. 提交成功 → 显示成功提示 → 1.2 秒后自动跳转到对应列表页
6. 提交失败 → alert 提示检查 Mock 服务

### API 方法

在原有 GET 方法基础上，为每个业务类型补充了 POST 方法：

```typescript
// trade.ts 示例
export function addTrade(data: Omit<TradeItem, 'id'>) {
  return http.post<TradeItem>('/trades', data)
}
```

数据提交时 id 由 JSON Server 自动生成，无需手动指定。

### 页面联动

- 发布成功后 @ 路由自动跳转到对应列表页（/trade、/lost-found、/group-buy、/errand）
- 列表页在 `onMounted` 中重新请求数据，可看到刚发布的信息
- db.json 文件中会新增一条记录

### AI 辅助生成内容

1. **PublishView.vue 表单结构** — AI 帮助生成了类型切换标签、条件渲染字段、表单提交函数框架
2. **FormField.vue 组件** — AI 生成了组件的基本结构

4. **POST API 方法** — AI 生成了四个业务类型的 addXxx() 函数

### 人工调整内容

1. **表单字段调整**：AI 生成的字段偏多（包含一些本项目不需要的字段），手动删除了"运费""品牌""规格"等不必要字段
2. **校验规则精简**：AI 对所有字段都加了校验，手动去掉了非必填字段的校验（如 eventTime、urgent 等）
3. **提交逻辑重构**：AI 将四个 API 调用放在一起，手动拆分为类型判断分支，使逻辑更清晰
4. **跳转路径修正**：AI 生成的跳转路径与实际路由不匹配，手动修正为 /trade、/lost-found 等
5. **类型问题修复**：AI 生成的部分代码存在 TypeScript 类型问题，手动补充了类型定义和断言

### 审查结论

AI 生成的代码在模板结构和基础逻辑上有参考价值，但字段设计、校验规则和业务路径需要人工根据项目实际情况调整。关键的业务逻辑（如提交后跳转、数据字段映射）必须经过人工审查。
